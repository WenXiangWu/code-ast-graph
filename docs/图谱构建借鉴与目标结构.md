# 图谱构建借鉴点与目标结构

> 参照《快手AI_Coding对比与借鉴》，梳理构建过程中的可借鉴点，以及当前/目标图结构

---

## 一、构建过程中可借鉴的点

### 1.1 代码层级显性化（构建阶段）

**在解析/转换阶段增加**：

| 借鉴点 | 实现方式 | 落地位置 |
|--------|----------|----------|
| arch_layer 属性 | 根据 `@Controller`、`@Service`、`@Repository`、`@Mapper` 推断 | `_create_type_entity` 或 scanner 提取类注解时 |
| 包路径推断 | `*.controller`、`*.service`、`*.dao`、`*.impl` 等 | Type 节点 metadata 或新增 `arch_layer` |
| 层级规则 | 定义 Controller→Service→DAO 的允许依赖方向 | 查询层做规则校验，构建时仅存储属性 |

**代码示意**（在 transformer / scanner 中）：

```python
# 根据注解推断 arch_layer
def _infer_arch_layer(cls: Dict) -> str:
    for ann in cls.get('annotations', []):
        name = ann.get('name', '') or ann.get('fqn', '')
        if 'Controller' in name or 'RestController' in name: return 'Controller'
        if 'Service' in name: return 'Service'
        if 'Repository' in name or 'Mapper' in name: return 'DAO'
    # 包路径兜底
    pkg = (cls.get('package') or '').lower()
    if 'controller' in pkg: return 'Controller'
    if 'service' in pkg and 'impl' not in pkg: return 'Service'
    if 'dao' in pkg or 'mapper' in pkg: return 'DAO'
    return 'Other'

# 在 Type 实体的 metadata 中存储
metadata['arch_layer'] = _infer_arch_layer(cls)
```

### 1.2 方法级 CALLS 边（构建阶段）

**现状**：calls 只生成 Type→Type 的 DEPENDS_ON，未生成 Method→Method 的 CALLS。

**借鉴**：快手以方法为节点，建立方法间调用边，便于路径查询和影响分析。

**实现**：在 transformer 中，对每个 call：

- 解析 caller_method_signature（当前方法）
- 解析 callee_method_signature（被调用方法，需 resolve target 类 + method 名）
- 若两者都能解析，创建 `(Method)-[:CALLS]->(Method)`

**难点**：callee 可能跨类（`a.foo()` 中 a 的类型需从字段/变量推导），解析成本较高。可先做**同文件、静态/显式调用**，再逐步扩展。

---

### 1.3 字段类型依赖（构建阶段）

**现状**：Field 有类型，但未建立 Type→Type 的 DEPENDS_ON（仅通过方法调用产生）。

**借鉴**：`private UserService userService` 应产生 `Type -[DEPENDS_ON]-> UserService`。

**实现**：在字段转换时，对 `field['type']` 做 FQN 解析，创建 `(Type)-[:DEPENDS_ON]->(Type)`（去重，避免与 calls 重复）。

---

### 1.4 方法返回类型 / 参数类型依赖（构建阶段）

**现状**：有 `(Method)-[:RETURNS]->(Type)`、`(Method)-[:HAS_PARAMETER]->(Parameter)`，Parameter 有 type，但未转为 Type 级 DEPENDS_ON。

**借鉴**：方法签名中的类型也是依赖，应参与依赖图。

**实现**：对每个 Method 的 return_type、参数类型，创建 `(Type)-[:DEPENDS_ON]->(Type)`（调用方 Type 依赖被引用类型）。

---

### 1.5 成对逻辑标注（构建阶段，可选）

**借鉴**：`listXxx` 与 `countXxx` 成对，在图谱中显式关联。

**实现**：扫描时识别成对方法（命名规则或注解），创建 `(Method)-[:PAIRED_WITH]->(Method)` 或类似关系，供影响分析使用。

---

### 1.6 target 解析增强（构建阶段）

**现状**：`call['target']` 可能是简单名，未通过 import 解析为 FQN。

**借鉴**：快手从字节码能精确定位类型，AST 需依赖 import 解析。

**实现**：在创建 DEPENDS_ON 时，优先用 `import_map` 将 target 解析为 FQN，再写入；无法解析则跳过，避免噪音节点。

## 二、当前图结构（构建完成后）

### 2.1 节点类型与属性

| 节点类型 | 标识字段 | 主要属性 | 来源 |
|----------|----------|----------|------|
| Project | name | path, scanned_commit_id, scanned_at | scan_service |
| Package | fqn | name | scanner 解析 |
| Type | fqn | name, file_path, project, package, kind... | 类声明 |
| Method | signature | name, return_type, class_fqn, parameters... | 方法声明 |
| Field | signature | name, type, class_fqn... | 字段声明 |
| Parameter | signature | name, type, position | 方法参数 |
| Annotation | fqn | name | 注解 |
| MQTopic | name + mq_type | group_id | MQ 监听/发送 |
| Table | name | entity_fqn | Mapper |

### 2.2 边类型与方向

| 边类型 | 起点 | 终点 | 含义 |
|--------|------|------|------|
| CONTAINS | Project | Package/Type | 项目包含 |
| CONTAINS | Package | Type | 包包含类 |
| DECLARES | Type | Method/Field | 类声明方法/字段 |
| DEPENDS_ON | Type | Type | 类依赖（目前仅来自方法调用） |
| EXTENDS | Type | Type | 继承 |
| IMPLEMENTS | Type | Type | 实现接口 |
| RETURNS | Method | Type | 返回类型 |
| HAS_PARAMETER | Method | Parameter | 方法参数 |
| ANNOTATED_BY | Type/Method/Field | Annotation | 注解 |
| DUBBO_CALLS | Type | Type | Dubbo 调用 |
| DUBBO_PROVIDES | Type | Type | Dubbo 服务实现 |
| LISTENS_TO_MQ | Method | MQTopic | MQ 监听 |
| SENDS_TO_MQ | Method | MQTopic | MQ 发送 |
| MAPPER_FOR_TABLE | Type | Table | Mapper 与表 |

**说明**：模型中存在 `CALLS`，但当前**没有** `(Method)-[:CALLS]->(Method)`，只有 Type 级 DEPENDS_ON。

### 2.3 当前图示意（简化）

```
(Project)-[:CONTAINS]->(Package)-[:CONTAINS]->(Type)
                              (Type)-[:DECLARES]->(Method)
                              (Type)-[:DECLARES]->(Field)
                              (Type)-[:DEPENDS_ON]->(Type)   ← 来自方法调用
                              (Type)-[:EXTENDS]->(Type)
                              (Type)-[:IMPLEMENTS]->(Type)
                              (Type)-[:DUBBO_CALLS]->(Type)
(Method)-[:RETURNS]->(Type)
(Method)-[:HAS_PARAMETER]->(Parameter)
(Method)-[:LISTENS_TO_MQ]->(MQTopic)
(Method)-[:SENDS_TO_MQ]->(MQTopic)
(Type)-[:MAPPER_FOR_TABLE]->(Table)
```

## 三、目标图结构（借鉴快手后）

### 3.1 节点增强

| 节点 | 新增/调整 | 说明 |
|------|------------|------|
| Type | arch_layer | Controller/Service/DAO/Helper/Other |
| Type | （可选）layer_priority | 用于分层规则校验 |
| Method | 保持不变 | 已有，需增加 CALLS 边 |
| （可选） | Constructor | 构造器单独建模，供影响分析 |

### 3.2 边增强

| 边类型 | 起点 | 终点 | 说明 |
|--------|------|------|------|
| CALLS | Method | Method | 新增：方法调用方法 |
| DEPENDS_ON | Type | Type | 扩产：字段类型、返回值、参数类型也产生 |
| PAIRED_WITH | Method | Method | 新增（可选）：成对逻辑如 list/count |

### 3.3 目标图示意

```
(Project)-[:CONTAINS]->(Package)-[:CONTAINS]->(Type {arch_layer})
    |
    +-- (Type)-[:DECLARES]->(Method)
    +-- (Type)-[:DECLARES]->(Field)
    +-- (Type)-[:DEPENDS_ON]->(Type)     ← 方法调用 + 字段类型 + 方法签名
    +-- (Type)-[:EXTENDS]->(Type)
    +-- (Type)-[:IMPLEMENTS]->(Type)
    +-- (Type)-[:DUBBO_CALLS]->(Type)
    |
(Method)-[:CALLS]->(Method)             ← 新增
(Method)-[:RETURNS]->(Type)
(Method)-[:HAS_PARAMETER]->(Parameter)
(Method)-[:PAIRED_WITH]->(Method)       ← 可选
(Method)-[:LISTENS_TO_MQ]->(MQTopic)
...
```

## 四、构建流程中的借鉴落地顺序

| 阶段 | 借鉴点 | 优先级 | 工作量 |
|------|--------|--------|--------|
| 解析 | 提取 arch_layer（注解+包路径） | P0 | 小 |
| 转换 | target 解析为 FQN（import_map） | P0 | 小 |
| 转换 | 字段类型 → DEPENDS_ON | P0 | 小 |
| 转换 | Method -[:CALLS]-> Method | P0 | 中 |
| 转换 | 方法签名类型 → DEPENDS_ON | P1 | 小 |
| 转换 | 成对逻辑 PAIRED_WITH | P1 | 中 |
| 存储 | 确保 arch_layer 写入 Type 节点 | P0 | 小 |
| 查询 | 分层视图、规则校验 API | P1 | 中 |

---

## 五、构建完成后的图能力

完成上述增强后，图可支持：

1. **分层视图**：按 arch_layer 过滤，展示 Controller→Service→DAO 分层结构  
2. **方法级路径**：沿 CALLS 做 shortestPath、影响范围分析  
3. **成对影响**：通过 PAIRED_WITH 做 list/count 等成对改动的联动分析  
4. **依赖来源区分**：DEPENDS_ON 可扩展 metadata 记录来源（call/field/return/param）  
5. **AI 图检索**：为 Agent 提供结构化 Context，支持上下游、影响范围、修改路径等查询  

---

## 六、总结

| 维度 | 当前 | 目标（借鉴快手） |
|------|------|------------------|
| 节点 | Project, Package, Type, Method, Field... | 增加 arch_layer 等属性 |
| 依赖边 | 仅 Type→Type（方法调用） | Type→Type（调用+字段+签名）+ Method→Method CALLS |
| 层级 | 无 | arch_layer 显式化 |
| 方法级能力 | 无 | CALLS、路径、影响分析 |
| 成对逻辑 | 无 | PAIRED_WITH 可选 |

构建阶段的核心改动：**在解析/转换时补全层级、依赖来源和方法级 CALLS**，存储层按现有模型扩展即可，查询层再基于新结构提供分层视图和路径分析。
