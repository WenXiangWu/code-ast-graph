# 架构分层识别规则 (arch_layer)

## 概述

在 `scanner_v2.py` 中,通过 `_infer_arch_layer` 方法为每个 Class 和 Interface 节点推断其在架构中的层次。这个属性存储在节点的 `arch_layer` 属性中。

## 识别规则

### 1. Mapper 层
**优先级**: 最高

**识别条件**:
- 类/接口被 `@Mapper` 注解修饰
- 类名以 `Mapper` 结尾
- 包路径包含 `mapper`
- 继承了 MyBatis 的 Mapper 接口

**示例**:
```java
@Mapper
public interface UserMapper extends BaseMapper<User> {
    // ...
}
```

**arch_layer**: `Mapper`

---

### 2. Controller 层
**优先级**: 高

**识别条件**:
1. **Spring MVC Controller**:
   - 类被 `@Controller` 或 `@RestController` 注解修饰

2. **Dubbo 服务入口**:
   - 类被 `@DubboService` 或 `@Service`(dubbo) 注解修饰
   - 且实现了 `*RemoteService` 接口

**示例**:
```java
// Spring MVC Controller
@RestController
@RequestMapping("/api/user")
public class UserController {
    // ...
}

// Dubbo 服务入口
@DubboService
public class NobleRemoteServiceImpl implements NobleRemoteService {
    // ...
}
```

**arch_layer**: `Controller`

---

### 3. Service 层
**优先级**: 中

**识别条件**:
1. **注解识别**:
   - 类被 `@Service` 注解修饰
   - 且不是 Dubbo 服务 (排除 `@DubboService`)

2. **包路径识别**:
   - 包路径包含 `service`
   - 且不包含 `impl` (避免与实现类混淆)

**示例**:
```java
// Spring Service
@Service
public class UserService {
    // ...
}

// 接口 (通过包路径识别)
// package com.example.service;
public interface IUserService {
    // ...
}
```

**arch_layer**: `Service`

---

### 4. Manager 层
**优先级**: 中

**识别条件**:
1. **类名识别**:
   - 类名以 `Manager` 结尾

2. **包路径识别**:
   - 包路径包含 `manager`

**示例**:
```java
// Manager 类
public class UserManager {
    @Resource
    private UserService userService;
    // ...
}

// 或者通过包路径
// package com.example.manager;
public class AccountManager {
    // ...
}
```

**arch_layer**: `Manager`

**说明**: Manager 层通常用于:
- 聚合多个 Service 的调用
- 处理复杂的业务逻辑编排
- 事务边界控制

---

### 5. DAO 层
**优先级**: 中

**识别条件**:
1. **注解识别**:
   - 类被 `@Repository` 注解修饰

2. **包路径识别**:
   - 包路径包含 `dao`

**示例**:
```java
// Spring Repository
@Repository
public class UserDao {
    // ...
}

// 或者通过包路径
// package com.example.dao;
public class AccountDao {
    // ...
}
```

**arch_layer**: `DAO`

**说明**: DAO 层与 Mapper 层的区别:
- **Mapper**: MyBatis 的 Mapper 接口,直接映射 SQL
- **DAO**: 数据访问对象,可能包含更复杂的数据访问逻辑

---

### 6. Entity 层
**优先级**: 中

**识别条件**:
1. **注解识别**:
   - 类被 `@Entity` 注解修饰 (JPA)

2. **包路径识别**:
   - 包路径包含 `entity`
   - 或包路径包含 `domain`

**示例**:
```java
// JPA Entity
@Entity
@Table(name = "user")
public class User {
    @Id
    private Long id;
    // ...
}

// 或者通过包路径
// package com.example.entity;
public class Account {
    // ...
}
```

**arch_layer**: `Entity`

---

### 7. Helper/Util 层
**优先级**: 低

**识别条件**:
- 类名以 `Helper` 结尾
- 或类名以 `Util` 结尾

**示例**:
```java
public class StringHelper {
    public static String trim(String str) {
        // ...
    }
}

public class DateUtil {
    public static Date parse(String dateStr) {
        // ...
    }
}
```

**arch_layer**: `Helper`

**说明**: 这类工具类通常会被业务类过滤规则过滤掉,因为它们没有注入字段。

---

### 8. Other 层
**优先级**: 最低 (默认)

**识别条件**:
- 不符合以上任何规则的类

**示例**:
```java
// DTO
public class UserDTO {
    private String name;
    // ...
}

// Request
public class CreateUserRequest {
    private String username;
    // ...
}

// Response
public class ApiResponse<T> {
    private int code;
    private T data;
    // ...
}
```

**arch_layer**: `Other`

**说明**: 这类非业务核心类通常会被业务类过滤规则过滤掉。

---

## 识别优先级

识别按以下顺序进行,一旦匹配就返回:

1. **Mapper** (最高优先级)
2. **Controller**
3. **Service**
4. **Manager**
5. **DAO**
6. **Entity**
7. **Helper**
8. **Other** (默认)

## 代码实现

```python
def _infer_arch_layer(
    self,
    class_name: str,
    package_name: str,
    annotations: List[Dict],
    is_mapper: bool,
    is_dubbo_service: bool,
    implements: List[str]
) -> str:
    """推断架构层次"""
    # Mapper (最高优先级)
    if is_mapper:
        return 'Mapper'
    
    # Controller
    ann_names = [a['name'] for a in annotations]
    if any(a in ann_names for a in ['Controller', 'RestController']):
        return 'Controller'
    if is_dubbo_service:
        # Dubbo 服务实现类,检查是否实现 RemoteService
        if any('RemoteService' in impl for impl in implements):
            return 'Controller'
    
    # Service
    if 'Service' in ann_names and not is_dubbo_service:
        return 'Service'
    if 'service' in package_name.lower() and 'impl' not in package_name.lower():
        return 'Service'
    
    # Manager
    if class_name.endswith('Manager'):
        return 'Manager'
    if 'manager' in package_name.lower():
        return 'Manager'
    
    # DAO
    if any(a in ann_names for a in ['Repository']):
        return 'DAO'
    if 'dao' in package_name.lower():
        return 'DAO'
    
    # Entity
    if 'Entity' in ann_names:
        return 'Entity'
    if 'entity' in package_name.lower() or 'domain' in package_name.lower():
        return 'Entity'
    
    # Helper/Util
    if class_name.endswith('Helper') or class_name.endswith('Util'):
        return 'Helper'
    
    # 默认
    return 'Other'
```

## 使用场景

### 1. 技术方案生成
根据 `arch_layer` 可以:
- 识别调用链路的层次结构
- 判断是否跨层调用
- 生成分层的技术方案文档

### 2. 代码分析
- 统计各层的类数量
- 分析层次间的依赖关系
- 检测架构违规 (如 Controller 直接调用 Mapper)

### 3. 图谱查询
```cypher
// 查询所有 Controller 层的类
MATCH (c:Class {arch_layer: 'Controller'})
RETURN c.fqn, c.name

// 查询 Controller -> Service -> Manager 的调用链
MATCH (c:Class {arch_layer: 'Controller'})-[:CALLS*]->(s:Class {arch_layer: 'Service'})-[:CALLS*]->(m:Class {arch_layer: 'Manager'})
RETURN c.name, s.name, m.name
```

## 扩展建议

如果需要支持更多的架构层次,可以在 `_infer_arch_layer` 方法中添加新的识别规则,例如:

- **Facade**: 外观层,聚合多个 Service
- **Gateway**: 网关层,API 网关
- **Adapter**: 适配器层,对接外部系统
- **Converter**: 转换器层,对象转换
- **Validator**: 校验器层,参数校验

## 注意事项

1. **识别顺序很重要**: 优先级高的规则会先匹配,避免误判
2. **Dubbo 服务特殊处理**: 实现 `*RemoteService` 的 Dubbo 服务被识别为 Controller 层
3. **包路径识别**: 使用 `lower()` 进行大小写不敏感匹配
4. **业务类过滤**: Helper/Util/Other 层的类如果没有注入字段,会被业务类过滤规则过滤掉
