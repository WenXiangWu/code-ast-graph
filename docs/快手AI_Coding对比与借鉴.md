# 快手「Context Is All You Need」 vs Code AST Graph 对比与借鉴

> 基于《Context Is All You Need：快手后端AI Coding的实践与思考》与 Code AST Graph 当前设计的对比分析

## 一、快手方案核心要点

### 1.1 问题定位

- **PRD ≠ 可执行的工程语言**：AI 缺少后端改造边界、入口位置、数据来源、缓存/异步影响等
- **隐式复杂度**：RPC、事务、binlog、Kafka、ES 等逻辑藏在 PRD 之外
- **AI 困境**：能改局部函数，但无法把握「这一行会牵动哪些地方」

### 1.2 两阶段演进

| 阶段 | 目标 | 做法 |
|------|------|------|
| **Context** | 让 AI 看到系统全貌 | 代码层级显性化 + 代码元素图谱 + Neo4j |
| **Control** | 让 AI 稳定可控执行 | Multi-Agent + 代码驱动（非 Prompt 驱动） |

### 1.3 码图框架：三件事

1. **代码如何分层**：Controller、Service、DAO、Helper 的职责与允许交互关系
2. **代码元素如何关联**：方法调用谁、字段被谁依赖、构造器影响
3. **需求沿何路径传播**：一次改动在调用链/数据流上的影响范围

### 1.4 技术实现

| 维度 | 快手做法 |
|------|----------|
| **数据来源** | 编译后**字节码**，非源码 AST |
| **图谱粒度** | 节点：字段、方法、构造器（类内元素）<br>边：调用、引用、继承、依赖 |
| **存储** | Neo4j 图数据库 |
| **AI 使用方式** | AI 生成 Cypher 查询 → 图检索 → 获取上下游、影响范围、修改路径 |
| **领域知识** | DeepWiki + LLM 解析仓库结构、业务术语、工程规范 → 向量化存储 |

## 二、与 Code AST Graph 的对比

### 2.1 数据来源与粒度

| 项目 | Code AST Graph | 快手 |
|------|----------------|------|
| **来源** | 源码 AST（javalang） | 编译后字节码 |
| **优势** | 无需编译、支持未完成项目 | 调用关系更准（含反射、动态） |
| **节点** | Project, Package, Type, Method, Field | 字段、方法、构造器 |
| **当前展示** | 主要为 Type - Type 依赖 | 方法级调用、字段依赖 |

### 2.2 代码层级

| 项目 | Code AST Graph | 快手 |
|------|----------------|------|
| **显式分层** | ❌ 无 | ✅ Controller/Service/DAO/Helper 显性化 |
| **层级规则** | ❌ 无 | ✅ 定义哪些层可交互、哪些改动有扩散效应 |
| **层级推断** | 部分（通过注解/路径可推断） | 显式建模 |

### 2.3 图谱能力

| 能力 | Code AST Graph | 快手 |
|------|----------------|------|
| Type 依赖 | ✅ DEPENDS_ON | ✅ |
| Method 调用 | ✅ 有 CALLS，但查询层以 Type 为主 | ✅ 方法级调用图谱 |
| 字段引用 | ⚠️ Field 有存储，引用关系有限 | ✅ 字段被谁依赖 |
| 构造器 | ❌ 未显式建模 | ✅ |
| Dubbo/MQ/Mapper | ✅ 有 | ✅ |
| 层级/构件视图 | ❌ | ✅ |

### 2.4 查询与 AI 集成

| 项目 | Code AST Graph | 快手 |
|------|----------------|------|
| **查询 API** | REST：按项目、起始类、深度、过滤模式 | AI 生成 Cypher 进行图检索 |
| **典型查询** | 调用图、影响面分析、数据库表 | 上下游、影响范围、修改路径 |
| **AI 集成** | 无 | 图检索 + 代码检索 + 语义检索 → 编码计划 |

## 三、可借鉴点与落地建议

### 3.1 代码层级显性化（高价值）

**现状**：我们有 Type、Method、Field，但未显式标注 Controller/Service/DAO/Helper 等分层。

**借鉴**：
- 按**注解**推断层级：`@Controller`、`@Service`、`@Repository`、`@Mapper` 等
- 按**包路径**推断：`*.controller`、`*.service`、`*.dao`、`*.mapper`
- 在 Neo4j 的 Type 节点上增加 `layer` 或 `arch_layer` 属性

**实现示意**：
```python
# 扫描时根据注解/包路径设置
type_metadata["arch_layer"] = "Controller" | "Service" | "DAO" | "Helper" | ...
```

**价值**：支持「分层视图」「分层规则校验」，为后续 AI 检索提供结构化信息。

---

### 3.2 方法级调用图与路径查询（高价值）

**现状**：已有 Method 和 calls，但查询层主要产出 Type 级 `DEPENDS_ON` 图。

**借鉴**：
- 增加 **方法级调用图** 查询：`(Method)-[:CALLS]->(Method)`
- 支持 **路径查询**：给定两个方法，返回最短调用路径（Cypher `shortestPath`）
- 支持 **影响范围**：从某方法出发，沿 CALLS 边能到达的所有方法

**实现方向**：
- 在 `neo4j_querier` 中增加 `get_method_call_graph(project, start_method?, max_depth)`
- 增加 `get_call_path(project, from_method, to_method)`
- 扩展 `analyze_impact` 到方法粒度

**价值**：解决「漏改」——列表查询与总数统计成对出现，需要方法级路径才能发现遗漏。

---

### 3.3 字节码 vs AST 的权衡（中价值）

**现状**：纯 AST 解析，无法覆盖反射、动态调用等。

**借鉴**：
- 快手从**字节码**构建图谱，调用关系更完整
- 可作为**可选增强**：对已编译项目，用 ASM/Javassist 解析 class 作为补充

**落地建议**：
- 短期：继续以 AST 为主，在现有基础上补全方法调用边
- 中期：对 Maven/Gradle 项目，在构建后解析 bytecode，与 AST 结果融合

---

### 3.4 概要设计与编码计划（体系参考）

**现状**：无 AI 执行流程，纯人工使用图谱。

**借鉴**：
- 概要设计：人拆模块边界、画 mermaid 流程图、标关键参考类
- 编码计划：由 SubAgent 基于图检索生成伪代码级计划，再由 Copilot 执行

**落地建议**：
- 图谱可作为 **Context 供给层**，为未来的 AI Agent 提供图检索能力
- 预留接口：`graph_search(project, query_type, params)` → 返回结构化依赖/路径数据
- 与 MCP、Cursor Rules 等结合：把图谱查询封装为工具，供 AI 调用

---

### 3.5 隐式依赖与成对逻辑（业务借鉴）

**案例**：列表查询与总数统计成对出现，AI 只改了一处导致分页错误。

**借鉴**：
- 在图谱中标注 **成对/成组** 关系：例如同 Controller 内「列表方法」与「总数方法」的关联
- 或通过命名/注解规则推断：`listXxx` ↔ `countXxx`

**落地**：可增加 `method_pair` 或 `logical_group` 类型关系，用于影响分析时联动提示。

---

## 四、落地优先级建议

| 优先级 | 项 目 | 预估工作量 | 说明 |
|--------|-------|------------|------|
| P0 | 代码层级显性化 | 小 | 注解/包路径推断，存到 Type 节点 |
| P0 | 方法级调用图 API | 中 | 利用已有 Method、calls，增加查询与路径能力 |
| P1 | 影响范围到方法级 | 中 | 基于 CALLS 做方法级 analyze_impact |
| P1 | 成对逻辑标注 | 中 | list/count 等成对方法的关联 |
| P2 | 字节码补充 | 大 | 可选，作为增强路径 |
| P2 | AI 图检索接口 | 中 | 为 Agent 提供结构化检索能力 |

---

## 五、总结

| 维度 | Code AST Graph 现状 | 快手启示 |
|------|---------------------|----------|
| 数据来源 | AST | 字节码可作增强 |
| 粒度 | Type 为主 | 方法、字段、构造器需加强 |
| 层级 | 无 | 层级显性化是核心 |
| 图谱能力 | 基础具备 | 方法级路径、影响范围需完善 |
| AI 集成 | 无 | 图谱作 Context 供给，可规划 |

**核心结论**：快手方案强调「Context 是后端 AI Coding 的决定性因素」，而**代码层级 + 方法级调用图谱**是 Context 的两大支柱。我们已有 Type 依赖与 Method 存储，重点可放在：

1. **补全方法级调用图能力**，支撑路径与影响分析；
2. **显式化代码层级**，支持分层视图与规则校验；
3. **预留图检索接口**，为后续 AI Agent 提供结构化 Context。
