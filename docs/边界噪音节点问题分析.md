# 图谱边界噪音节点问题分析

## 一、问题现象

查询出的依赖图中，边界点（叶子节点）存在大量低价值或无效信息，如：
- **Response**：HTTP 响应封装类
- **VO**：值对象
- **Code**：通常为枚举或状态码类
- **log**：日志相关（Logger 或 Log 类）

这些节点对理解业务调用关系帮助有限，反而干扰阅读。

---

## 二、构建流程：节点与边如何产生

### 2.1 数据流

```
源码 (*.java)
    ↓ [javalang AST 解析]
JavaASTScanner._parse_java_file()
    ↓ 提取
classes, methods, fields, calls, imports
    ↓ [JavaASTTransformer 或 _store_to_neo4j]
Neo4j: Type 节点 + DEPENDS_ON 边
```

### 2.2 DEPENDS_ON 边的唯一来源：方法调用

当前 **DEPENDS_ON 只来自方法调用**（`MethodInvocation`）：

```python
# scanner.py: _extract_method_calls
# 对每个 x.y() 形式的调用：
call_info = {
    'caller_class': current_class,   # 当前类（调用方）
    'callee': node.member,            # 方法名（如 info, ok, getData）
    'target': qualifier_str,          # 接收对象/类名（如 log, Response, Code）
}
```

存储时解析逻辑（transformer 与 scanner 的 _store_to_neo4j 类似）：

```python
callee_class = call.get('target') or import_map.get(call['callee'])
# target 优先：Response.ok() → target="Response"
#             log.info()   → target="log"
#             code.getValue() → target="code"
if callee_class:
    MERGE (caller)-[:DEPENDS_ON]->(callee)  # 创建边和 callee 节点
```

### 2.3 为何会出现 Response、Code、log、VO

| 调用形式 | target | 结果 |
|----------|--------|------|
| `Response.ok()` | Response | 创建 Type(Response) 和 DEPENDS_ON |
| `Code.SUCCESS` | Code | 创建 Type(Code) 和 DEPENDS_ON |
| `log.info(...)` | log | 若可解析为类则创建（如某 Log 类） |
| `vo.getId()` | vo | 变量名，多无法解析；若解析到 VO 类则创建 |
| `Result.success(data)` | Result | 创建 Type(Result) 和 DEPENDS_ON |

只要能在 `target` 或 `import_map` 中解析出类名，就会创建 DEPENDS_ON 和对应 Type 节点。  
**不会区分**：结构依赖（如 Service→Mapper）vs 基础设施/包装类依赖（如 →Response、→Code、→Logger）。

## 三、过滤机制：为何 moderate 仍会看到噪音

### 3.1 查询时过滤（noise_filter）

```python
# config/noise_filter.py
# moderate 模式过滤：JDK + 工具库包 + Utils/Helper 等后缀
# 但不过滤：Response, VO, Code, Request 等
# 这些仅在 strict 模式下过滤
```

| 过滤项 | loose | moderate | strict |
|--------|-------|----------|--------|
| JDK (java.lang.*) | ✅ | ✅ | ✅ |
| 工具库 (slf4j, guava...) | ❌ | ✅ | ✅ |
| Utils/Helper 后缀 | ❌ | ✅ | ✅ |
| Response/VO/Code/Request | ❌ | ❌ | ✅ |

因此 **moderate 下** Response、VO、Code 等会保留。

### 3.2 过滤发生在查询阶段

过滤在 `neo4j_querier._do_get_call_graph` 中执行：先查 Neo4j，再对每条边用 `filter_func` 判断是否保留。若默认使用 `moderate`，这些类型不会被滤掉。

---

## 四、根本原因总结

1. **构建阶段**：DEPENDS_ON 仅依赖方法调用，且不区分业务调用与基础设施调用。  
   - `Response.ok()`、`Code.xxx`、`log.info()` 等都会建立 DEPENDS_ON。

2. **target 解析**：以 `qualifier`（target）为主，可能得到简单类名，未充分结合 `import_map` 解析为 FQN，易产生不完整或重复的 Type 节点。

3. **未利用的类型信息**：  
   - 字段类型（如 `Logger log`）  
   - 返回值类型（如 `Response<X>`）  
   - 参数类型  
   这些都可能引入 DEPENDS_ON，但当前仅依赖方法调用的 target。

4. **过滤策略**：moderate 不包含 Response、VO、Code 等，只有 strict 才会过滤，而 strict 可能过滤过猛。

---

## 五、改进方向

### 5.1 查询层增强（短期，推荐）

- 将 Response、Code、VO、Request、Result 等加入 **moderate** 的过滤规则。  
- 在 `DTO_SUFFIXES` 中补充更多常见“胶水类型”，并在 moderate 中启用对它们的过滤。

### 5.2 构建层：调用类型区分（中期）

- 识别 **静态调用**（如 `Response.ok()`、`Code.SUCCESS`）与 **实例调用**（如 `log.info()`）。  
- 对静态调用，判断是否属于已知的“基础设施/包装类”，若是则可选择：  
  - 不建 DEPENDS_ON，或  
  - 使用单独的关系类型（如 `WRAPS`），在查询时按需过滤。

### 5.3 构建层：类型解析增强

- 优先用 `import_map` 将 target 解析为 FQN。  
- 对 `log`、`vo` 等变量，结合字段/参数类型推导实际类型，再决定是否建 DEPENDS_ON。

### 5.4 构建层：依赖类型分级（长期）

- 将 DEPENDS_ON 分为：  
  - **核心依赖**：Service→Mapper、Controller→Service 等  
  - **基础设施依赖**：Logger、Response、Code 等  
- 查询时支持按依赖类型过滤（如只显示核心依赖）。

---

## 六、快速可落地的修改

1. **noise_filter**：在 moderate 中增加对 Response、Code、VO、Request、Result、Log 等常见噪音类型的过滤（可先用类名后缀或包名前缀）。
2. **前端**：将默认过滤模式由 moderate 调整为 strict，或在 moderate 中应用扩展后的规则，观察噪音是否明显减少。
3. **构建**：在创建 DEPENDS_ON 前，先判断 target 是否属于预设的“噪音类”列表，若是则跳过该边的创建。
