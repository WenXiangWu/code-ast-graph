# 技术方案图谱设计方案 v2.0

> 基于「开通贵族」等业务场景，从零设计的图数据库 Schema  
> 核心：仅追踪注入依赖调用链，过滤非业务类

---

## 一、核心设计原则

### 1.1 调用追踪规则

| 规则 | 说明 | 示例 |
|------|------|------|
| **注入字段四件套** | 仅关注 @Reference、@DubboReference、@Resource、@Autowired 修饰的字段 | `@Resource private NobleManager nobleManager;` |
| **Dubbo 调用** | 通过 @Reference/@DubboReference 字段发起 → DUBBO_CALLS | `nobleRemoteService.openNoble()` |
| **内部调用** | 通过 @Resource/@Autowired 字段发起 → CALLS | `nobleManager.changeNobleInfo()` |
| **忽略非注入** | 参数校验、对象组装、非注入字段调用一律忽略 | `Objects.isNull(request)` 不建边 |
| **业务类过滤** | 类内部无注入字段 → 非核心业务类 → 不建 Type 节点 | Util、DTO、Request 等被过滤 |

### 1.2 终点识别

| 终点类型 | 识别规则 |
|----------|----------|
| **Mapper** | @Mapper 注解 OR 接口名以 Mapper 结尾 OR 包名含 mapper + 参数含 entity |
| **Dubbo 接口** | 被 @Reference/@DubboReference 引用的接口 |
| **Table** | 从 Mapper 接口名、XML、注解等推断表名 |

## 二、节点设计

### 2.1 核心节点

| 标签 | 主键 | 关键属性 | 说明 |
|------|------|----------|------|
| **Repo** | `name` | path, git_url, branch, scanned_at, commit_id | git-repos 下的仓库（如 yuer-chatroom-service） |
| **Module** | `fqn` | name, path, repo_name | Maven/Gradle 子模块（如 yuer-chatroom-core） |
| **Type** | `fqn` | name, file_path, arch_layer, kind, has_injection, repo, module | 类/接口（**仅保留有注入字段或被注入的类**） |
| **Method** | `signature` | name, class_fqn, return_type, line_start, line_end, is_entry | 方法 |
| **Field** | `signature` | name, type_fqn, class_fqn, injection_type | 字段（**仅保留注入字段**） |

**Type.has_injection**：布尔值，标识该类是否有注入字段（用于快速过滤）。  
**Field.injection_type**：`Reference` | `DubboReference` | `Resource` | `Autowired` | `null`（仅注入字段有值）。  
**Method.is_entry**：标识是否为 RPC/HTTP 入口（有 @MobileAPI 等注解）。

### 2.2 技术方案专用节点

| 标签 | 主键 | 关键属性 | 说明 |
|------|------|----------|------|
| **Table** | `name` | schema, entity_fqn, mapper_fqn | 数据库表 |
| **MQTopic** | `name` + `mq_type` | mq_type, group_id | Kafka/RocketMQ topic（保留现有逻辑） |
| **Job** | `fqn` | name, job_type, cron_expr, class_fqn, method_signature | Aries 定时/延时任务 |
| **RpcEndpoint** | `path` + `method` | path, http_method, service_name, method_signature | HTTP/Dubbo 对外接口 |

**Job**：从 `@AriesCronJobListener`、`@AriesDelayJobListener`、`@Scheduled` 等注解解析。  
**RpcEndpoint**：从 `@MobileAPI(path=...)`、`@PostMapping`、`@RequestMapping` 等解析。

### 2.3 索引

```cypher
CREATE INDEX type_fqn IF NOT EXISTS FOR (t:Type) ON (t.fqn);
CREATE INDEX type_has_injection IF NOT EXISTS FOR (t:Type) ON (t.has_injection);

CREATE INDEX method_signature IF NOT EXISTS FOR (m:Method) ON (m.signature);
CREATE INDEX method_class_fqn IF NOT EXISTS FOR (m:Method) ON (m.class_fqn);
CREATE INDEX method_is_entry IF NOT EXISTS FOR (m:Method) ON (m.is_entry);
CREATE INDEX field_injection_type IF NOT EXISTS FOR (f:Field) ON (f.injection_type);
CREATE INDEX table_name IF NOT EXISTS FOR (t:Table) ON (t.name);
CREATE INDEX mq_topic_name IF NOT EXISTS FOR (mt:MQTopic) ON (mt.name);
CREATE INDEX job_fqn IF NOT EXISTS FOR (j:Job) ON (j.fqn);
CREATE INDEX rpc_endpoint_path IF NOT EXISTS FOR (r:RpcEndpoint) ON (r.path);
```

## 三、边设计

### 3.1 结构边

| 边类型 | 起点 | 终点 | 属性 | 说明 |
|--------|------|------|------|------|
| CONTAINS | Repo | Module | - | 仓库包含子模块 |
| CONTAINS | Module | Type | - | 模块包含类 |
| DECLARES | Type | Method | - | 类声明方法 |
| DECLARES | Type | Field | - | 类声明字段（**仅注入字段**） |
| EXTENDS | Type | Type | - | 继承 |
| IMPLEMENTS | Type | Type | - | 实现接口 |

### 3.2 调用边（核心）

| 边类型 | 起点 | 终点 | 属性 | 说明 |
|--------|------|------|------|------|
| **DUBBO_CALLS** | Type | Type（接口） | field_name, method_name | 通过 @Reference/@DubboReference 字段调用 Dubbo 接口 |
| **DUBBO_PROVIDES** | Type | Type（接口） | - | @DubboService 实现类提供 Dubbo 服务 |
| **CALLS** | Method | Method | via_field | 通过 @Resource/@Autowired 字段发起的内部调用 |

**DUBBO_CALLS.method_name**（可选）：记录调用的方法名（如 `openNoble`），便于方法级追踪。  
**CALLS.via_field**：记录通过哪个字段调用（如 `nobleManager`），便于溯源。

### 3.3 技术方案专用边

| 边类型 | 起点 | 终点 | 属性 | 说明 |
|--------|------|------|------|------|
| **MAPPER_FOR_TABLE** | Type | Table | - | Mapper 接口映射表（保留现有逻辑） |
| **ENTITY_FOR_TABLE** | Type | Table | - | Entity/JPA 映射表（可选扩展） |
| **LISTENS_TO_MQ** | Method | MQTopic | group_id | 消费 MQ（保留现有逻辑） |
| **SENDS_TO_MQ** | Method | MQTopic | - | 发送 MQ（保留现有逻辑） |
| **EXECUTES_JOB** | Method | Job | - | 方法执行定时任务 |
| **EXPOSES** | Method | RpcEndpoint | - | 方法对外暴露 RPC/HTTP 入口 |

### 3.4 移除的边

| 边类型 | 原用途 | 移除原因 |
|--------|--------|----------|
| DEPENDS_ON | Type→Type，所有方法调用产生 | 被 DUBBO_CALLS 和 CALLS 替代，避免噪音 |

## 四、构建流程

### 4.1 第一遍扫描：识别注入字段

```python
# 伪代码
injected_fields = {}  # class_fqn -> { field_name: { anno, type_fqn } }

for cls in classes:
    cls_injected = {}
    for field in cls.fields:
        for anno in field.annotations:
            if anno in ['Reference', 'DubboReference', 'Resource', 'Autowired']:
                cls_injected[field.name] = {
                    'annotation': anno,
                    'type_fqn': resolve_type_fqn(field.type, imports, package)
                }
    
    if cls_injected:
        injected_fields[cls.fqn] = cls_injected
```

**输出**：`injected_fields` 字典，供后续调用解析使用。

### 4.2 第二遍扫描：提取调用关系

```python
# 伪代码
for method in methods:
    caller_class = method.class_fqn
    caller_method_sig = method.signature
    
    for call in method.body.calls:  # MethodInvocation
        qualifier = call.qualifier  # 如 nobleRemoteService
        callee_method = call.member  # 如 openNoble
        
        # 检查 qualifier 是否为注入字段
        if caller_class in injected_fields:
            field_info = injected_fields[caller_class].get(qualifier)
            if field_info:
                anno = field_info['annotation']
                target_type = field_info['type_fqn']
                
                if anno in ['Reference', 'DubboReference']:
                    # 创建 DUBBO_CALLS (Type -> Type)
                    create_edge(
                        type='DUBBO_CALLS',
                        from_type=caller_class,
                        to_type=target_type,
                        metadata={'field_name': qualifier, 'method_name': callee_method}
                    )
                elif anno in ['Resource', 'Autowired']:
                    # 创建 CALLS (Method -> Method)
                    callee_method_sig = f"{target_type}.{callee_method}(...)"
                    create_edge(
                        type='CALLS',
                        from_method=caller_method_sig,
                        to_method=callee_method_sig,
                        metadata={'via_field': qualifier}
                    )
            # else: 非注入字段，忽略
```

### 4.3 业务类过滤

在创建 Type 节点前，检查：

```python
if cls.fqn not in injected_fields and not is_injected_by_others(cls.fqn):
    # 无注入字段 且 不被其他类注入 → 非业务核心类
    skip_class(cls)
```

**is_injected_by_others**：检查是否有其他类的注入字段类型为 `cls.fqn`（如 NobleManager 被 NobleController 注入）。

**保留**：
- 有注入字段的类（如 NobleController、NobleManager）
- 被注入的类（如 NobleService、NobleMapper）
- Mapper 接口（即使无注入字段，也是终点）
- Dubbo 接口（被 @Reference 引用）

**过滤**：
- Util、Helper、DTO、Request、Response 等无注入字段且不被注入的类

### 4.4 RPC 入口识别

```python
for method in methods:
    for anno in method.annotations:
        if anno.name in ['MobileAPI', 'PostMapping', 'RequestMapping', 'GetMapping']:
            path = extract_path_from_annotation(anno)  # 如 /official/open/noble
            http_method = extract_http_method(anno)    # GET/POST
            
            create_node(
                type='RpcEndpoint',
                id=f"{path}#{method.name}",
                properties={'path': path, 'http_method': http_method, 'method_signature': method.signature}
            )
            create_edge(
                type='EXPOSES',
                from_method=method.signature,
                to_endpoint=f"{path}#{method.name}"
            )
            
            # 标记方法为入口
            method.is_entry = True
```

### 4.5 Mapper 识别（含无 @Mapper）

当前逻辑已覆盖：

1. 接口有 `@Mapper` 注解
2. 接口名以 `Mapper` 结尾（如 BlackRoomMatchRuleConfigMapper）
3. 包名含 `mapper` 且方法参数含 entity/domain 类型

**补充**：确保 `BlackRoomMatchRuleConfigMapper` 等无 @Mapper 的接口在 `InterfaceDeclaration` 分支被识别（约 scanner.py 507-547 行）。

### 4.6 MQ 处理（保留现有）

保留现有的 MQ 监听和发送逻辑：

- `@KafkaListener`、`@RocketMQMessageListener` → LISTENS_TO_MQ
- `kafkaTemplate.send()`、`rocketMQTemplate.send()` → SENDS_TO_MQ
- 创建 MQTopic 节点，Method-[:LISTENS_TO_MQ|SENDS_TO_MQ]->MQTopic

### 4.2 扩散 Cypher

```cypher
// 从入口方法开始扩散
MATCH (entry:RpcEndpoint {path: '/official/open/noble'})
MATCH (entry)<-[:EXPOSES]-(m1:Method)

// 找到实现类（通过 IMPLEMENTS）
MATCH (m1)<-[:DECLARES]-(t1:Type)-[:IMPLEMENTS]->(iface:Type)

MATCH path = (dubbo_impl)-[:DECLARES]->(m2:Method)-[:CALLS*0..10]->(mn:Method)

// 找到 Mapper
OPTIONAL MATCH (mn)<-[:DECLARES]-(mapper:Type)-[:MAPPER_FOR_TABLE]->(table:Table)

RETURN path, table
```

## 六、过滤逻辑

### 6.1 类级过滤

**保留条件**（满足任一）：

1. `has_injection = true`（有注入字段）
2. 被其他类注入（在 `injected_fields` 的 `type_fqn` 中出现）
3. 是 Mapper（@Mapper 或 *Mapper 接口）
4. 是 Dubbo 接口（被 @Reference 引用）
5. 是 Entity（被 Mapper 引用）

**过滤**：不满足以上条件的类（Util、DTO、Request、Response、Code、ExceptionCode 等）。

### 6.2 调用级过滤

**仅建边**：调用的 qualifier 在 `injected_fields` 中。

**忽略**：

- 静态方法调用（如 `Objects.isNull()`、`StringUtils.isEmpty()`）
- 链式调用（如 `request.getUid()`）
- 非注入字段调用

## 七、查询路径设计

### 7.1 从入口扩散到底层

```cypher
// 输入: RPC 入口 path
MATCH (endpoint:RpcEndpoint {path: $entry_path})
MATCH (endpoint)<-[:EXPOSES]-(entry_method:Method)

// 扩散: CALLS + DUBBO_CALLS + DUBBO_PROVIDES
MATCH path = (entry_method)-[:CALLS|DECLARES*0..1]-(:Type)-[:DUBBO_CALLS*0..3]->(dubbo_iface:Type)
OPTIONAL MATCH (dubbo_impl:Type)-[:DUBBO_PROVIDES]->(dubbo_iface)
MATCH (dubbo_impl)-[:DECLARES]->(m:Method)-[:CALLS*0..10]->(end_method:Method)

// 收集终点: Mapper、MQ
OPTIONAL MATCH (end_method)<-[:DECLARES]-(mapper:Type)-[:MAPPER_FOR_TABLE]->(table:Table)
OPTIONAL MATCH (end_method)-[:LISTENS_TO_MQ|SENDS_TO_MQ]->(mq:MQTopic)

RETURN path, table, mq
```

```python
):
    if isinstance(node, javalang.tree.MethodInvocation):
        qualifier = str(node.qualifier) if node.qualifier else None
        callee = node.member
        
        # 检查 qualifier 是否为注入字段
        injection_type = None
        target_type_fqn = None
        
        if qualifier and current_class in injected_fields:
            field_info = injected_fields[current_class].get(qualifier)
            if field_info:
                injection_type = field_info['annotation']
                target_type_fqn = field_info['type_fqn']
        
        # 仅记录注入调用
        if injection_type:
            call_info = {
                'caller_class': current_class,
                'caller_method': current_method_signature,
                'qualifier': qualifier,
                'callee_method': callee,
                'injection_type': injection_type,
                'target_type_fqn': target_type_fqn
            }
            calls.append(call_info)
    
    # 递归...
```

### 8.3 边创建逻辑

在 `_store_to_neo4j` 或 `transformer.py` 中：

```python
for call in calls:
    injection_type = call.get('injection_type')
    
    if injection_type in ['Reference', 'DubboReference']:
        # 创建 DUBBO_CALLS (Type -> Type)
        self.client.execute_write("""
            MATCH (caller:Type {fqn: $caller_class})
            MERGE (service:Type {fqn: $service_interface})
            SET service.name = $service_name
            MERGE (caller)-[:DUBBO_CALLS {
                field_name: $field_name,
                method_name: $method_name
            }]->(service)
        """, {
            'caller_class': call['caller_class'],
            'service_interface': call['target_type_fqn'],
            'service_name': call['target_type_fqn'].split('.')[-1],
            'field_name': call['qualifier'],
            'method_name': call['callee_method']
        })
    
    elif injection_type in ['Resource', 'Autowired']:
        # 创建 CALLS (Method -> Method)
        callee_signature = f"{call['target_type_fqn']}.{call['callee_method']}(...)"
        
        self.client.execute_write("""
            MATCH (caller:Method {signature: $caller_method})
            MERGE (callee:Method {signature: $callee_method})
            SET callee.name = $callee_name,
                callee.class_fqn = $callee_class
            MERGE (caller)-[:CALLS {via_field: $via_field}]->(callee)
        """, {
            'caller_method': call['caller_method'],
            'callee_method': callee_signature,
            'callee_name': call['callee_method'],
            'callee_class': call['target_type_fqn'],
            'via_field': call['qualifier']
        })
```

### 8.4 业务类过滤

在创建 Type 节点前：

```python
# 第一步：收集所有被注入的类型
injected_types = set()
for cls_fields in injected_fields.values():
    for field_info in cls_fields.values():
        injected_types.add(field_info['type_fqn'])

# 第二步：过滤类
for cls in classes:
    has_injection = cls['fqn'] in injected_fields
    is_injected = cls['fqn'] in injected_types
    is_mapper = cls.get('is_mapper', False)
    is_dubbo_interface = cls['fqn'] in dubbo_interfaces  # 从 dubbo_references 收集
    
    if not (has_injection or is_injected or is_mapper or is_dubbo_interface):
        # 过滤掉：Util、DTO、Request、Response 等
        continue
    
    # 创建 Type 节点
    cls['has_injection'] = has_injection
    create_type_node(cls)
```

### 9.2 从 code-index-demo 到图谱的流程

1. **code-index-demo 语义检索**：「开通贵族」→ 返回类名（GuardManager、NobleController 等）
2. **提取项目与类**：从结果中提取 `project`（如 yuer-chatroom-service）和 `class_name`
3. **图谱查询入口**：
   ```cypher
   MATCH (r:Repo {name: $project})
   MATCH (r)-[:CONTAINS*]->(t:Type)
   WHERE t.name = $class_name OR t.fqn CONTAINS $class_name
   RETURN t.fqn
   ```
4. **扩散调用链**：以 `t.fqn` 为起点，沿 CALLS + DUBBO_CALLS + DUBBO_PROVIDES 扩散
5. **提取表、MQ、Job**：从扩散结果中查询 MAPPER_FOR_TABLE、LISTENS_TO_MQ、SENDS_TO_MQ、EXECUTES_JOB
6. **回 code-index-demo**：用扩散得到的类名做 grep 精确检索，拉取完整代码
