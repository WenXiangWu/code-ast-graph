# 技术方案导向的图数据库设计

> 面向 PRD → 技术方案 生成场景，从零设计的图数据库 Schema 与查询模型  
> 项目来源：git-repos 下的所有仓库

---

## 一、设计目标

技术方案生成需要从 PRD 功能点出发，快速回答：

| 问题 | 图需支持的能力 |
|------|----------------|
| 改这个功能，代码动到哪？ | 项目 → 类 → 方法，调用链扩散到底层 |
| 涉及哪些表？ | 类/方法 → Mapper → Table，Entity → Table |
| 涉及哪些 MQ？ | 生产/消费 topic、对应类与方法 |
| 涉及哪些定时/延时任务？ | Job 类、调度配置、关联方法 |
| 改 A 会影响谁？ | 上游调用、下游依赖、影响面评估 |
| 分层结构如何？ | Controller / Service / DAO 定位 |

**核心原则**：以「可执行级技术方案」的产出为导向，反向设计节点、边和查询路径。

## 二、节点设计（Node Labels）

### 2.1 结构节点

| 标签 | 主键 | 关键属性 | 用途 |
|------|------|----------|------|
| **Repo** | `path` / `name` | path, name, git_url, branch, scanned_at, commit_id | git-repos 下的一个仓库，作为顶层聚合 |
| **Module** | `fqn` | name, path, repo | Maven/Gradle 子模块（如 chatroom-trade-core） |
| **Type** | `fqn` | name, file_path, arch_layer, kind, project | 类/接口/枚举 |
| **Method** | `signature` | name, class_fqn, return_type, line_start, line_end | 方法 |
| **Field** | `signature` | name, type_fqn, class_fqn | 字段 |

**说明**：

- `Repo`：与 git-repos 一一对应，`Module` 为子模块粒度，便于与 code-index-demo 的项目名对齐
- `Type.arch_layer`：`Controller` / `Service` / `Manager` / `DAO` / `Mapper` / `Entity` / `Helper` / `Other`，由注解与包路径推断
- `Type.kind`：`CLASS` / `INTERFACE` / `ENUM`

### 2.2 技术方案专用节点

| 标签 | 主键 | 关键属性 | 用途 |
|------|------|----------|------|
| **Table** | `name` | schema, entity_fqn | 数据库表 |
| **MQTopic** | `name` + `mq_type` | mq_type, group_id | Kafka/RocketMQ topic |
| **Job** | `fqn` | name, job_type, cron_expr, class_fqn | 定时/延时任务 |
| **RpcEndpoint** | `fqn` | service_name, method_name | Dubbo/HTTP 对外接口 |
| **BusinessFeature** | `id` | name, keywords, prd_ref | 业务功能点（可选，用于关联） |

**说明**：

- `Job`：从 `@AriesCronJobListener`、`@AriesDelayJobListener`、Aries 等注解解析，`job_type` 如 `scheduled` / `delayed`
- `RpcEndpoint`：Dubbo 接口 + 方法、HTTP Controller 的 path + method，便于「入口」定位

### 2.3 索引建议

```cypher
CREATE INDEX type_fqn IF NOT EXISTS FOR (t:Type) ON (t.fqn);
CREATE INDEX type_arch_layer IF NOT EXISTS FOR (t:Type) ON (t.arch_layer);
CREATE INDEX method_signature IF NOT EXISTS FOR (m:Method) ON (m.signature);
CREATE INDEX method_class_fqn IF NOT EXISTS FOR (m:Method) ON (m.class_fqn);
CREATE INDEX table_name IF NOT EXISTS FOR (t:Table) ON (t.name);
CREATE INDEX mq_topic_name IF NOT EXISTS FOR (mt:MQTopic) ON (mt.name);
CREATE INDEX repo_name IF NOT EXISTS FOR (r:Repo) ON (r.name);
```

## 三、边设计（Relationship Types）

### 3.1 结构边

| 边类型 | 起点 | 终点 | 属性 | 含义 |
|--------|------|------|------|------|
| CONTAINS | Repo | Module | - | 仓库包含子模块 |
| CONTAINS | Module | Type | - | 模块包含类 |
| DECLARES | Type | Method | - | 类声明方法 |
| DECLARES | Type | Field | - | 类声明字段 |
| DEPENDS_ON | Type | Type | source? | 类型依赖（call/field/return/param） |
| CALLS | Method | Method | source? | 方法调用方法 |
| EXTENDS | Type | Type | - | 继承 |
| IMPLEMENTS | Type | Type | - | 实现接口 |
| DUBBO_CALLS | Type | Type | field_name | Dubbo 调用 |
| DUBBO_PROVIDES | Type | Type | - | Dubbo 实现 |

**DEPENDS_ON.source**（可选）：`call` | `field` | `return` | `param`，用于区分依赖来源，便于过滤噪音。

### 3.2 技术方案专用边

| 边类型 | 起点 | 终点 | 属性 | 含义 |
|--------|------|------|------|------|
| MAPPER_FOR_TABLE | Type | Table | - | Mapper 映射表 |
| ENTITY_FOR_TABLE | Type | Table | - | Entity/JPA 映射表 |
| LISTENS_TO_MQ | Method | MQTopic | - | 消费 MQ |
| SENDS_TO_MQ | Method | MQTopic | - | 发送 MQ |
| EXECUTES_JOB | Method | Job | - | 方法执行定时任务 |
| EXPOSES | Method | RpcEndpoint | path? | 方法对外暴露 RPC/HTTP |

### 3.3 可选增强边

| 边类型 | 起点 | 终点 | 含义 |
|--------|------|------|------|
| PAIRED_WITH | Method | Method | list/count 等成对逻辑 |
| SAME_FEATURE | Type | Type | 同业务功能（人工或规则标注） |

## 四、查询路径设计（以技术方案生成为中心）

### 4.1 调用链扩散（CodeResearcher 核心）

**输入**：起始类名/方法名（来自 code-index-demo 语义检索）  
**输出**：完整调用图（类、方法、边），扩散到底层无新节点为止

**路径**：

```
(Type {name: 'GuardManager'})
  -[:DECLARES]->(Method)-[:CALLS]->(Method)<-[:DECLARES]-(Type)
  -[:DEPENDS_ON]->(Type)
  -[:DUBBO_CALLS]->(Type)
```

**策略**：优先用 `Method-[:CALLS]->Method` 做方法级扩散；若无 CALLS，退化为 `Type-[:DEPENDS_ON]->Type`。

### 4.2 表关联提取

**路径**：

```
(扩散得到的 Type)
  -[:MAPPER_FOR_TABLE|ENTITY_FOR_TABLE]->(Table)
```

**补充**：若 Type 为 Mapper 实现类，通过 IMPLEMENTS 找到接口，再查 MAPPER_FOR_TABLE。

### 4.3 MQ 拓扑提取

**路径**：

```
(扩散得到的 Method)
  -[:LISTENS_TO_MQ|SENDS_TO_MQ]->(MQTopic)
```

**输出**：`{ topic, mq_type, producers: [class.method], consumers: [class.method] }`。

### 4.4 Job 提取

**路径**：

```
(扩散得到的 Type)-[:DECLARES]->(Method)-[:EXECUTES_JOB]->(Job)
或
(Job)<-[:EXECUTES_JOB]-(Method)<-[:DECLARES]-(Type)
```

**备选**：若无 EXECUTES_JOB，通过 `Type-[:ANNOTATED_BY]->(@Scheduled)` 等注解反向查找。

### 4.5 分层视图

**路径**：

```
MATCH (t:Type)
WHERE t.arch_layer IN ['Controller','Service','DAO']
  AND (t)-[:DEPENDS_ON|CALLS*1..5]->(target)
RETURN t, target
```

按 `arch_layer` 过滤，用于架构合规校验和展示分层结构。

### 4.6 影响面分析

**路径**：

```
(修改目标 Type/Method)
  <-[:DEPENDS_ON|CALLS*1..N]-(调用方)
```

双向分析：下游影响（改 A 影响谁）、上游溯源（谁调用了 A）。

## 五、与 git-repos 的映射

| git-repos 结构 | 图建模 |
|----------------|--------|
| `git-repos/official-room-pro-service/` | Repo(name: official-room-pro-service, path: ...) |
| `git-repos/official-room-pro-service/official-room-service-manager/` | Module(fqn: ...-manager, repo: official-room-pro-service) |
| `chatroom-trade-core` 子模块 | Module，其下 Type 的 project 可存 repo 或 module 名 |

**项目名对齐**：code-index-demo 的项目名可能是 repo 或 module，建议图中同时存储 `Type.repo` 和 `Type.module`，查询时做兼容匹配。

## 六、构建流程建议

### 6.1 扫描顺序

1. 遍历 git-repos 下每个目录，创建 Repo 节点
2. 识别 Maven/Gradle 子模块，创建 Module 节点
3. 对每个 Module 的 Java 源码：Type、Method、Field
4. 解析调用、依赖、注解，创建 CALLS、DEPENDS_ON、LISTENS_TO_MQ、SENDS_TO_MQ、MAPPER_FOR_TABLE、EXECUTES_JOB 等边
5. 推断 arch_layer，写入 Type 节点

### 6.2 增量更新

- 基于 Git diff 检测变更文件
- 只重新解析变更文件，增量更新节点和边
- 删除已移除文件对应的节点

### 6.3 跨仓库依赖

- 当 DEPENDS_ON/CALLS 的目标类属于其他 repo 时，创建占位 Type 或通过跨 repo 扫描补全
- Dubbo 接口的提供者可能在另一仓库，需支持跨 repo 的 DUBBO_PROVIDES 解析

## 七、查询 API 建议（供 Agent 调用）

| API | 输入 | 输出 | 用途 |
|-----|------|------|------|
| `diffuse_call_graph` | repo/module, start_classes[], max_depth | nodes, edges | 调用链扩散 |
| `get_tables_for_types` | type_fqns[] | tables[], mapper_map | 表关联 |
| `get_mq_topology` | repo/module? | topics, producers, consumers | MQ 拓扑 |
| `get_jobs_for_types` | type_fqns[] | jobs[] | Job 列表 |
| `get_layer_view` | repo, layer? | types by layer | 分层视图 |
| `analyze_impact` | type/method, direction | affected_nodes | 影响面 |
| `get_rpc_endpoints` | repo? | endpoints[] | 入口列表 |

## 八、与现有设计的差异总结

| 维度 | 原设计 | 技术方案导向设计 |
|------|--------|------------------|
| 顶层 | Project | Repo + Module（对齐 git-repos） |
| 调用 | Type-DEPENDS_ON-Type | Method-CALLS-Method + Type-DEPENDS_ON-Type |
| 层级 | 无 | Type.arch_layer 显式 |
| 表 | Mapper→Table | Mapper→Table + Entity→Table |
| Job | 无 | Job 节点 + EXECUTES_JOB 边 |
| 入口 | 无 | RpcEndpoint 节点 |
| 查询 | 通用 | 面向技术方案生成场景的专用 API |
| 依赖来源 | 不区分 | DEPENDS_ON.source 可选区分 |

| 阶段 | 内容 | 优先级 |

## 十、总结

本设计以「从 PRD 功能点 → 可执行技术方案」为主线，重新梳理节点、边和查询路径：

1. **Repo/Module** 对齐 git-repos，与 code-index-demo 项目名可映射  
2. **arch_layer** 支持分层视图和架构校验  
3. **Method-CALLS-Method** 支持方法级调用链扩散  
4. **Table、MQTopic、Job** 等专用节点直接服务于技术方案中的表、MQ、定时任务描述  
5. **专用查询 API** 覆盖调用链扩散、表提取、MQ 拓扑、Job 提取、影响面分析等场景  

按上述优先级分阶段实施，可逐步形成面向技术方案生成的图能力。
